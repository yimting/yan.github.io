Hbase--漫漫之路

发表于 2019-04-22 | 更新于 2019-05-20 | 分类于 Learn

摘要：HBase是一种 “NoSQL” 数据库. NoSQL是一个通用词表示数据库不是RDBMS ，后者支持 SQL 作为主要访问手段。

介绍

Hbase是什么

​ HBase是一种构建与Hadoop架构之上的 NoSQL 数据库. NoSQL是一个通用词表示数据库不是RDBMS ，后者支持 SQL 作为主要访问手段。有许多种 NoSQL 数据库: BerkeleyDB 是本地 NoSQL 数据库例子, 而 HBase 是大型分布式数据库。 技术上来说, HBase 更像是”数据存储(Data Store)” 多于 “数据库(Data Base)”。因为缺少很多RDBMS特性, 如列类型，第二索引，触发器，高级查询语言等.

​ 然而, HBase 有许多特征同时支持线性化和模块化扩充。 HBase 集群通过增加RegionServers进行扩充。 它可以放在普通的服务器中。例如，如果集群从10个扩充到20个RegionServer，存储空间和处理容量都同时翻倍。 RDBMS 也能很好扩充， 但仅对一个点 - 特别是对一个单独数据库服务器的大小 - 同时，为了更好的性能，需要特殊的硬件和存储设备。
同时，HBase 是一个多行三列的存储方式(列存储方式)

第一列： RowKey
第二列： Properties
第三列： value

例如：

HBase 特性：

  • 强一致性读写: HBase 不是 “最终一致性(eventually consistent)” 数据存储. 这让它很适合高速计数聚合类任务。
  • 自动分片(Automatic sharding): HBase 表通过region分布在集群中。数据增长时，region会自动分割并重新分布。
  • RegionServer 自动故障转移
  • Hadoop/HDFS 集成: HBase 支持本机外HDFS 作为它的分布式文件系统。
  • MapReduce: HBase 通过MapReduce支持大并发处理， HBase 可以同时做源和目标.
  • Java 客户端 API: HBase 支持易于使用的 Java API 进行编程访问.
  • Thrift/REST API: HBase 也支持Thrift 和 REST 作为非Java 前端.
  • Block Cache 和 Bloom Filters: 对于大容量查询优化， HBase支持 Block Cache 和 Bloom Filters。
  • 运维管理: HBase提供内置网页用于运维视角和JMX 度量.

启动Hbase后，可以打开Web界面

Hbase的架构是什么

自我理解：

在 Hbase中拥有多个客户端 （ region Server ），每一个 region Server 中负责表中一部分的数据(region)，
当你在查询的时候，会将查询出来的结果保存到 region Server 的缓存中，
每个 region Server 的数据范围是一个（ region ） ，当你使用的过程中 regionServer 出现宕机行为，如何处理 ？Hbase 会对于操作进行一个记录，保存到 HDFS 的 WAL log 日志中，这个WAL 日志会进行定时进行合并操作，

那么如果region Server比较多 通过谁来管理呢 （HMaster），HMaster 通过 Zookeeper 进行管理（ 不能直接管理 region Server ） region Server （小弟），
小弟去进行管理数据表

Client

  • 包含访问Hbase的接口并且维护cache从而加快对Hbase的访问

Zookeeper[^6]

  • 保证任何时候，集群中只有一个 Master
  • 维护哪些服务器处于活动状态且可用，并实时通知 Master。
  • Zookeeper 来保证共同的共享状态。
  • 存储 Hbase 的 Schema 和 Table 的元数据信息

Master

  • 为 region server 分配 region
  • 负责 region server 的负载均衡
  • 发现失效的 region server 并且进行重新分配 region
  • 管理 用户对于 表的增删改操作

RegionServer

  • 负责维护 region ， 处理 region 所产生的 IO 请求
  • 负责切分 在运行过程中 过大的 region

Hlog

  • HLog文件就是一个普通的Hadoop Sequence File，Sequence File 的Key是HLogKey对象，HLogKey中记录了写入数据的归属信息，除了table和 region名字外，同时还包括sequence number和timestamp，timestamp是” 写入时间”，sequence number的起始值为0，或者是最近一次存入文件系 统中sequence number。
  • HLog SequeceFile的Value是HBase的KeyValue对象，即对应HFile中的 KeyValue

Hbase的存储过程

在Hbase的架构中，存在如下几个组件操作

  1. 预写日志 ： WAL （write ahead log），所有的客户端进行写入操作的时候，会先将操作的日志写入到WAL中
  2. MeMStore ： 当缓存达到一定阈值的时候，会进行 flush操作，会将 MeMStore中数据， 溢写到 磁盘中（HDFS），也叫做 存储文件【 Store File （HFile^1）】
  3. 当溢写本地的 HFile 数量达到一行阈值的时候，会进行合并操作，合并成一个更大的 StoreFile ， 当StoreFile 大小达到一行阈值的时候，再次进行 split 操作，将 region 划分成两个
  4. HMaster 会把其中一个 Region 分发到其他的 RegionServer 上，从而实现 服务器的 负载均衡

对于Hbase 的删除， 是在删除的时候给 所要删除的数据打上标签， 在合并中进行删除（也就是说 合并操作中，不包含那些打上标记的数据）

Hbase怎样工作的

  • 读数据流程

Hbase中存在两张表[^3]

  1. 客户端首先访问 Zookeeper ， 从 meta 表中读取 region 的位置， 之后读取 meta表中的数据， meta 中又存储了用户表 region 信息

  2. 根据 namespace ，表名 和 rowkey 在 meta 表中找到对应的 region 信息

  3. 寻找到对应的这个 region

  4. 从 MeMStore 找数据， 如果不存在， 再到 StoreFile 上读取 （提高读取的效率）
  • 写数据流程
  1. Client 首先访问 Zookeeper ， 从 meta 表获取相对应的 region 信息 ， 之后找到 meta 表的数据
  2. namespace、表名、 rowkey 会 根据 meta 表的数据找到写入数据对应的 region 信息
  3. 找到相对应的 regionServer
  4. 将数据分别写到 HLog 和 MeMStore 一份
  5. MeMStore 达到一个阈值 后将数据 进行 刷新操作，合并成一个 StoreFile 文件[^4]
  6. 当 StoreFile 达到一定的阈值之后，进行Compact合并操作，合并成一个StoreFIle[^5]
  7. 当 合并后的 StoreFile 达到一定大小的时候，会将当前的 region 进行分割操作，并由HMaster 分散到其他的 RegionServer，实现负载均衡

Hbase有什么好处

  • HBase不适合所有问题.

首先，确信有足够多数据，如果有上亿或上千亿行数据，HBase是很好的备选。 如果只有上千或上百万行，则用传统的RDBMS可能是更好的选择。因为所有数据可以在一两个节点保存，集群其他节点可能闲置。

其次，确信可以不依赖所有RDBMS的额外特性 (e.g., 列数据类型, 第二索引, 事物,高级查询语言等.) 一个建立在RDBMS上应用，如不能仅通过改变一个JDBC驱动移植到HBase。相对于移植， 需考虑从RDBMS 到 HBase是一次完全的重新设计。

第三， 确信你有足够硬件。甚至 HDFS 在小于5个数据节点时，干不好什么事情 (根据如 HDFS 块备份数为 3), 还要加上一个 NameNode.

HBase 和 Hadoop/HDFS 的区别?

​ HDFS 是分布式文件系统，适合保存大文件。官方宣称它并非普通用途文件系统，不提供文件的个别记录的快速查询。 另一方面，HBase基于HDFS且提供大表的记录快速查找(和更新)。这有时可能引起概念混乱。 HBase 内部将数据放到索引好的 “存储文件(StoreFiles)” ，以便高速查询。存储文件位于 HDFS中。参考Chapter 5, 数据模型 和该章其他内容获取更多HBase如何归档的信息。
